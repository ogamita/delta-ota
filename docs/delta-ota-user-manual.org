* About this manual

This manual is for two audiences:

1. *Network administrators* who deploy and operate a delta-ota
   server for their organisation, decide who can publish releases,
   and manage client classifications.
2. *Software integrators* who ship a desktop application and want
   their users to receive over-the-air updates with bandwidth
   smaller than the full payload.

For internal engineering and architecture details see
[[file:ota-specifications.org][ota-specifications.org]] and [[file:operations.org][operations.org]] (the operator
runbook).  This document is the customer-facing reference.

* Concepts

** Software, release, blob, patch

A *software* product is identified by a short kebab-case name
(=schms=, =plugin-cao=).

A *release* is one immutable build of one software for one
=(operating system, architecture)= target — for example
=schms/windows-x86_64/3.4.1=.  Each release ships as:

- A *blob*: a deterministic =.tar= archive of the release tree,
  uploaded to the server.  The same source produces the same bytes,
  every time — this is what makes the binary deltas tight.
- A *manifest*: a small JSON document that describes the release
  and lists the patches available to it.  Every manifest is signed
  with the server's Ed25519 key.

A *patch* is a binary delta from one blob to another.  v1 uses
=bsdiff= patches (BSDIFF40 + bzip2).  Each patch is referenced from
the destination release's manifest.

** Channels and classifications

A *channel* is a named subset of releases visible to a class of
clients — e.g. =stable=, =beta=, =canary=, =internal=.

A *classification* is the equivalent label on the *client* side.
A client carrying classification =[stable, beta]= sees every
release tagged =stable= OR =beta=.

Classifications are assigned to a client when its install token is
minted.  This is how you ship beta builds to a known group of
testers without exposing them to the general public.

** Install token, bearer token

To pick up a software for the first time, an unmanaged client
needs an *install token* — short-lived, one-shot, minted by an
admin.  When the agent uses it, the server exchanges it for a
*bearer token* unique to that workstation.  Subsequent calls (list,
manifest, blob, patch downloads) carry the bearer; the install
token is consumed and discarded.

The bearer is persisted in the agent's =state.json= and is reused
for every later upgrade.  Rotating it requires re-running install
with a fresh install token.

* For the network administrator

** Hosting choices

Two supported topologies (see [[file:operations.org::*Deployment topologies][operations.org]] for detail):

- *Single dedicated host* — the simplest deployment.  SQLite
  catalogue, filesystem blob storage, systemd unit, Linux x86_64 or
  arm64.
- *Kubernetes cluster* — for cloud deployments behind a load
  balancer.  PostgreSQL catalogue, S3-compatible blob storage.
  (PG and S3 backends ship in a sub-phase between v1 and v1.1; v1
  is the single-host topology.)

The same container image (=registry.gitlab.com/ogamita/delta-ota/server=)
is published as a multi-arch manifest (linux/amd64 + linux/arm64)
and serves both topologies.  Storage and database backends are
selected in =ota.toml=.

** Network requirements

Inbound :

- *TCP/443 (HTTPS)* :: the public download port.  TLS termination
  is typically done at a reverse proxy (nginx, HAProxy, a cloud
  LB).  When TLS is offloaded, the OTA server speaks plain HTTP
  on the loopback so the kernel =sendfile= path stays active for
  blob downloads.
- *TCP/443 admin path* :: separate admin endpoints under
  =/v1/admin/...= require an admin bearer token.  Restrict at the
  proxy layer to known admin networks.  Since v1.4, the proxy can
  also forward a verified client certificate's Subject DN via a
  configurable header; pair with =[tls].admin_subjects= for a
  strict allowlist and =[tls].require_mtls= for the strictest
  bearer-AND-cert mode.  ADR-0009 has the design + nginx snippet.

Outbound :

- *DNS / NTP* :: standard.
- *Container registry* :: pulling the server image
  (=registry.gitlab.com/ogamita/...=).  No outbound dependency at
  runtime once the image is on disk.
- *Notification webhook* :: when =[notifications].webhook_url= is
  configured (v1.7+), the server POSTs upgrade-notification
  payloads to an operator-managed receiver.  See ADR-0012 +
  =tools/notifications/=.

The server makes *no other* outbound calls during normal
operation: publishing happens via inbound POST, builds happen
in-process, and clients pull.

** Capacity planning

The numbers below are conservative for a 2 GB-class software
product with a monthly release cadence and ~10 000 clients.

| Workload                        | Per-host figure                          |
|---------------------------------+------------------------------------------|
| Steady-state upgrade            | 20 MB median per client (1 % of blob)    |
| 10 000 simultaneous upgrades    | ~200 GB egress over ~80 s @ 2×10 GbE     |
| Cold install (first-time user)  | up to 2 GB transfer (rate-limited)       |
| Disk: per release archive       | ~ blob size                              |
| Disk: per active patch          | 5–50 MB median                           |
| RAM: patch worker               | ≈ 2 × blob size at build time            |
| CPU: ongoing                    | low; bursts on publish                   |

For larger fleets or faster cadences, scale storage first (blobs
add up over time) and add patch-build workers (sub-phase 5).

** Day-to-day operations

The full operator runbook lives in [[file:operations.org][operations.org]].  The actions
you'll perform most often:

*** Publish a release (developer or CI)

#+begin_src sh
ota-admin publish ./build/release/                  \
    --software=schms                                \
    --version=3.4.1                                 \
    --os=windows --arch=x86_64                      \
    --os-versions=10,11                             \
    --classifications=stable
#+end_src

*** Mint an install token for a new user

#+begin_src sh
curl -X POST -H "Authorization: Bearer $OTA_ADMIN_TOKEN"        \
     -H "Content-Type: application/json"                        \
     -d '{"classifications":["stable"],"ttl_seconds":900}'      \
     https://ota.example.com/v1/admin/install-tokens
#+end_src

The response carries a one-shot =install_token= valid for 15
minutes (configurable).  Hand it to the user via a one-time link
or pasted into your install-page URL.

*** Garbage-collect old releases

#+begin_src sh
curl -X POST -H "Authorization: Bearer $OTA_ADMIN_TOKEN"        \
     -H "Content-Type: application/json"                        \
     -d '{"min_user_count":0,"min_age_days":30}'                \
     https://ota.example.com/v1/admin/software/schms/gc
#+end_src

Idempotent.  Run weekly from cron / systemd timer.

Since v1.6 the GC is /reachability-aware/: it consults the
client-software snapshot (v1.5) to make sure dropping a release
doesn't strand any client on a no-patch upgrade path.  For each
client whose only upgrade path would be broken, the GC pre-builds
the minimum set of direct =current → target= patches before the
drop.  Three new body keys control this layer:

| Key                       | Default | Effect                                                  |
|---------------------------+---------+---------------------------------------------------------|
| =ensure_reachability=     | =true=  | Run the reachability layer.  =false= = pre-v1.6 behaviour. |
| =allow_blob_fallback=     | =false= | Accept full-blob fallback for affected clients instead of pre-building. |
| =max_built_edges=         | =50=    | Refuse the GC entirely when more edges would be needed. |

When the plan needs more edges than =max_built_edges=, the GC
returns =409= and drops nothing.  Re-run with =--max-built-edges=N=
to raise the cap, or =--allow-blob-fallback= to accept the
fallback for the affected clients.  ADR-0011 has the full design.

The CLI mirror is =ota-server gc --software=NAME= with the
matching =--no-ensure-reachability= / =--allow-blob-fallback=
/ =--max-built-edges= flags.

*** Inspect the fleet (v1.5+)

The admin stats catalogue exposes a small, curated set of named
SQL queries against the client-software snapshot and the event
log.  Two equivalent entry points:

#+begin_src sh
# HTTP: returns JSON.
curl -H "Authorization: Bearer $OTA_ADMIN_TOKEN" \
     "https://ota.example.com/v1/admin/stats/population-per-release?software=schms"

# CLI: returns an ASCII table.
ota-server stats population-per-release --software=schms
ota-server stats --help-stats                       # lists the catalogue
#+end_src

Available queries (each documented at =GET /v1/admin/stats=):

| Name                     | Use case                                       |
|--------------------------+------------------------------------------------|
| =population-per-release= | who is on X right now (the bread-and-butter)   |
| =fleet-summary=          | cross-software view                            |
| =stale-clients=          | clients that haven't reported in N days        |
| =gc-impact=              | dry-run view of which releases the GC would prune |
| =recent-events=          | time-bucketed transition rate                  |
| =upgrade-failure-rate=   | per (from, to) pair (HAVING attempts ≥ 5)      |
| =adoption-curve=         | cumulative-adoption curve per release          |
| =recovery-events=        | =doctor --recover= rate (quality / network)    |

Per-row result cap: 1000.  Operators wanting fuller dumps run
=sqlite3 db/ota.db= directly.  ADR-0010 covers the design.

*** Send an upgrade announcement (v1.7+)

When =[notifications].webhook_url= is configured in =ota.toml=,
every publish automatically fans out an upgrade notification to
clients on an older version of the same software.  Operators
can also trigger a fan-out manually:

#+begin_src sh
curl -X POST -H "Authorization: Bearer $OTA_ADMIN_TOKEN"        \
     -H "Content-Type: application/json"                        \
     -d '{"release_id":"schms/linux-x86_64/3.4.1","reason":"announce"}' \
     https://ota.example.com/v1/admin/software/schms/announce
#+end_src

=ota-server= ships *no* SMTP code; mail leaves the process via
an HTTP webhook.  Operators run a small receiver (samples in
=tools/notifications/=: SMTP, Slack) that translates our JSON
to the channel their organisation uses.  ADR-0012 covers the
payload schema + HMAC signature + retry semantics.

*** Verify storage integrity

#+begin_src sh
curl -X POST -H "Authorization: Bearer $OTA_ADMIN_TOKEN"        \
     https://ota.example.com/v1/admin/verify
#+end_src

Reports any artefact whose on-disk hash no longer matches its
content-addressed name (= disk corruption).

*** Inspect the audit log

#+begin_src sh
curl -H "Authorization: Bearer $OTA_ADMIN_TOKEN"                \
     https://ota.example.com/v1/admin/audit
#+end_src

Every admin write is recorded with identity + timestamp.

** Backup and restore

#+begin_src sh
sudo tools/backup.sh /var/lib/ota /var/backups/ota
sudo tools/restore.sh /var/backups/ota/ota-backup-XXXX.tar.gz /var/lib/ota
#+end_src

The catalogue plus all blobs/patches/manifests/keys are bundled
into a single tarball with a SHA-256 sidecar.  See [[file:operations.org::*Backup and disaster recovery][operations.org →
Backup]] for details.

** Key rotation

The server's Ed25519 manifest-signing key lives at
=$OTA_ROOT/etc/keys/=.  Rotation procedure with zero downtime is
documented in [[file:operations.org::*Key rotation][operations.org → Key rotation]].  Distribute the new
public key to clients *before* rotating, then re-sign every
manifest in place with =ota-server.workers:resign-manifests=.

* For the software integrator

You ship a desktop application; you want it to update itself in
the field.  delta-ota gives you two integration shapes:

1. *Shell out to =ota-agent='* (recommended).
2. *Link =libota='* into your application via the C ABI.

Pick (1) unless you have a specific reason to need in-process
progress callbacks during a long-running session of your app.

** Option 1 — shell out to =ota-agent=

The =ota-agent= binary is a single static executable per OS.
Bundle it next to your application, or place it on =PATH=.

A typical update flow from inside your application is:

#+begin_src sh
# In your application, on a daily timer or "Check for updates" menu:
ota-agent upgrade my-software --to=latest
case "$?" in
  0) # Update succeeded; restart the application.
     ;;
  1) # No update available, or transient error.  Fail soft.
     ;;
esac
#+end_src

For first-time installation by a user, point them at the URL of
the install page.  The page hands the user a one-shot install
token and the agent binary; the agent runs:

#+begin_src sh
ota-agent install my-software                                   \
    --server https://ota.example.com                            \
    --install-token $TOKEN                                      \
    --hwinfo "$(hostname)"
#+end_src

After this, the agent writes a =state.json= containing the
per-workstation bearer token; subsequent =upgrade= calls reuse
that bearer and require no further authentication.

The agent installs into =$OTA_HOME/<software>/= (default
=~/.ota/<software>/=) and exposes the active distribution as
=$OTA_HOME/<software>/current= — a symlink on POSIX, or, on
Windows non-admin/non-developer-mode installs, a =current.path=
text file holding the absolute path.

Since v1.3 the agent keeps its =<dst>.part= file across attempts:
if a 2 GB initial install drops at 1.9 GB, the next =install=
invocation reuses the partial bytes (after scanning the trailing-
zero tail and trimming back to the last committed byte), asks
the server for =Range: bytes=N-=, and resumes.  ADR-0008 has the
trailing-zero rationale (no per-chunk =fsync(2)= cost).  No
agent-side configuration is needed; resume is always on.

Your application launcher reads =$OTA_HOME/<software>/current=
and execs the binary inside.  See [[*Launcher patterns]].

*** Client-side state reporting (v1.5+)

After every successful =install= / =upgrade= / =revert= /
=doctor --recover=, the agent PUTs the per-(client, software)
state to =/v1/clients/me/software/<sw>=.  The server uses this
snapshot for an *exact* "who is on X right now" count -- which
the GC's =min_user_count= threshold now actually consults
(pre-v1.5 the check was structurally off because no agent
reported).

Reporting is best-effort: a network failure logs at warning
level but does NOT roll the local install back.  The local
state in =$OTA_HOME/<software>/state.json= is always the
authoritative record from the agent's perspective.

To opt out (sandboxed installs, privacy-conscious deployments),
set =OTA_DISABLE_REPORTING=1= in the agent's environment.  The
local install proceeds normally; only the server-side snapshot
is suppressed.  Note that opted-out clients don't count toward
the GC's =min_user_count= threshold -- operators should pair the
opt-out with =uncollectable= pins on releases they want to keep.

*** Notifications: registering an email (v1.7+)

Users can opt in to receive upgrade notifications:

#+begin_src sh
ota-agent set-email   my-software alice@example.com
ota-agent show-email  my-software
ota-agent unset-email my-software [alice@example.com]    # without addr, removes all
#+end_src

The agent does *not* prompt for email on install -- users invoke
=set-email= explicitly.  Address storage is plain-text in the
server's catalogue; deletion is one HTTP call via the
=unset-email= subcommand or =DELETE /v1/clients/me/email=
(GDPR right-to-deletion).

The actual mail dispatch happens in an operator-managed webhook
receiver, not in =ota-server=.  See ADR-0012 for the payload
schema and the sample receivers in =tools/notifications/=.

** Option 2 — link =libota= via the C ABI

Use this when your application needs to drive the upgrade in-
process (e.g. a long-running CAD plug-in that wants to display
progress in its own UI).

The C header is generated by the Go build:

#+begin_src
// libota.h — five exported functions:
extern char* ota_last_error(void);
extern char* ota_version(void);
extern int   ota_install(char* cfg_json, char* name, char* version);
extern int   ota_upgrade(char* cfg_json, char* name, char* version);
extern int   ota_revert (char* cfg_json, char* name);
#+end_src

Each function returns =0= on success, =-1= on failure; on failure
=ota_last_error()= returns a thread-safe pointer to the most
recent error string.  =cfg_json= is a JSON-encoded =Config=:

#+begin_src json
{
  "ServerURL":      "https://ota.example.com",
  "OTAHome":        "C:\\Users\\joe\\.ota",
  "TrustedPubKeys": ["a8e1...c4"],
  "InstallToken":   "first-time-only",
  "Hwinfo":         "joe-laptop",
  "FallbackRatio":  0.7
}
#+end_src

The library is published as:

| Platform        | Shared        | Static         | Header     |
|-----------------+---------------+----------------+------------|
| Linux amd64/arm64 | =libota.so=   | =libota.a=     | =libota.h= |
| macOS amd64/arm64 | =libota.dylib= | =libota.a=    | =libota.h= |
| Windows amd64   | =libota.dll=  | =libota.a=     | =libota.h= |

Bindings examples (Python and .NET) live under =examples/= in the
repository.

** Launcher patterns

To keep your application launching while OTA is upgrading the next
distribution in the background, use a *launcher shim*:

- The shim is a tiny binary that lives at a fixed path (the path
  the desktop shortcut or =Open With= association points to).
- On launch, it reads =$OTA_HOME/<software>/current= and execs the
  real binary inside.
- delta-ota's atomic-flip means the shim never sees a half-
  installed tree.

A reference implementation in Go (about 30 lines) ships under
=examples/launcher-shim/=.

** Trust and signing

Every manifest is signed with the server's Ed25519 key.  The agent
verifies the signature against a *trusted public-key set*:

- /Pinned/: pass =--trusted-pubkeys=KEY1,KEY2= or set
  =OTA_TRUSTED_PUBKEYS=.  The agent refuses any manifest not
  signed by one of these keys.
- /Trust-on-first-use/: when no trusted set is configured, the
  agent records the first key it sees in =state.json= and pins
  subsequent calls to that key.  A change of key (rotation, or a
  hostile server replacement) is rejected unless the user passes
  =--trust-new-key=.

For production deployments, ship your application with the
production key pinned.

** What to put in your release tarball

The release tree handed to =ota-admin publish= becomes the
=current= distribution on every client.  It should contain
everything your application needs at runtime — your binary, your
DLLs/dylibs, your resources — *and nothing else*.

Avoid:

- machine-specific paths
- timestamps inside files (the deterministic-tar layer takes care
  of file metadata, but if your build produces files that embed
  build dates, you'll defeat the delta compression)
- unstable file ordering (e.g. randomised obfuscation seeds)

The more deterministic your build, the smaller your patches.  A
good rule of thumb is: a release that recompiles your code without
touching a single source line should produce a < 1 %-of-blob
patch.
