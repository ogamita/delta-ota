#+TITLE: OTA — Implementation Plan
#+AUTHOR: Pascal Bourguignon
#+OPTIONS: toc:3 num:t
#+STARTUP: showall

This plan turns [[file:ota-specifications.org][ota-specifications.org]] into a delivery sequence.
Phases are *vertical slices*: each one ends on a working,
demonstrable system, not on a layer.

* Guiding principles

- /Contract first/ : the HTTP/JSON API, the manifest schema, and
  the =libota= C ABI are written down before code touches the
  network.  Anything else can be rewritten cheaply.
- /Boring first/ : SQLite + filesystem blobs + Woo + Go are the
  default; postpone PostgreSQL, S3, mTLS hardware tokens, automated
  channel promotion, and the recovery tool to phases where their
  cost is paid by a real need.
- /End-to-end early/ : phase 1 already does upload → patch →
  download → install on one OS, even though half the code is stubs.
  Every later phase keeps that pipeline green.
- /Test what hurts/ : determinism (blob byte-identity), atomicity
  (failure mid-install never breaks =current=), patch correctness
  (reconstructed blob hash matches manifest).  Integration tests run
  on real disk; unit tests stay narrow.

The =server/= and =admin/= subtrees are SBCL-only; =client/= is a
single Go module that builds three artefacts: the =libota= shared
library, the =libota.h= header, and the =ota-agent= CLI.

* External dependencies

The project is *self-contained*: the client ships as a single
static binary with no runtime dependencies, and the server builds
its own =bsdiff= and =xdelta3= helper binaries from vendored
sources rather than relying on what the host distro packages.
See [[file:ota-specifications.org::*Self-containment and vendoring][Self-containment and vendoring]] for the policy.

** Build-time, available from package managers (server only)

| Concern             | Library                           | Pin             | License       |
|---------------------+-----------------------------------+-----------------+---------------|
| HTTP server         | Woo (Clack adapter), libev        | latest stable   | LLGPL/MIT — verify per-component before pin |
| TLS (server)        | cl+ssl + system OpenSSL ≥ 3.0     | OS package      | MIT (cl+ssl) + OpenSSL (Apache-2.0) |
| JSON                | =jzon=                            | latest stable   | MIT           |
| SQL (server)        | cl-dbi + cl-sqlite + cl-postgres  | latest stable   | LLGPL / MIT   |
| Crypto/signing      | =ironclad=                        | latest stable   | BSD-3-Clause  |

These are normal Quicklisp / system dependencies pulled at build
time on the build host.  They are /not/ shipped in the client
binary.

** Vendored into the repo (committed under =vendor/= and =client/internal/vendor/=)

| Component                       | Upstream                       | License       | Lang     | Used by                                   |
|---------------------------------+--------------------------------+---------------+----------+-------------------------------------------|
| bsdiff / bspatch (client)       | =gabstv/go-bsdiff=             | MIT           | pure Go  | =libota= patch applier                    |
| bsdiff / bspatch (server)       | =mendsley/bsdiff=              | BSD-2-Clause  | C        | server patch builder (built as small CLI) |
| tar (server)                    | =sharplispers/archive=         | BSD-3-Clause  | Common Lisp | server ingestion / extraction tests   |

All licences verified permissive (MIT, BSD-2/3-Clause).  GPL/LGPL/
AGPL is /prohibited/ by the vendoring policy.

*v1 ships with =bsdiff= as the only patcher.*  An earlier draft
listed =jmacd/xdelta3= (Apache-2.0) but on actual import the v3.1.0
tree turned out to be *GPLv2-or-later*; see [[file:adr/0004-bsdiff-only-no-xdelta3.org][ADR-0004]].
=ably/vcdiff-go= (Apache-2.0 decoder-only) is also dropped because
without a permissive VCDIFF encoder it would be dead code.

The Go client uses the stdlib =archive/tar=, =crypto/ed25519=,
=crypto/sha256=, =net/http=, =crypto/tls= — no third-party
dependencies for these concerns, no =cgo=.

A versions matrix with pinned commit hashes and SHA-256 of each
imported tree is maintained in =docs/dependencies.org= and verified
by CI (see Phase 0).

* Phase 0 — Foundations & vendoring  (1 week)

Goal :: nothing runs yet, but the contracts, skeletons, /and the
vendored third-party sources/ are in place and verified.

** Skeletons and contracts

- [ ] =docs/= : this plan + ADR-0001 "stack split: SBCL server / Go
  client"; ADR-0002 "Woo + sendfile, not Hunchentoot"; ADR-0003
  "vendor permissively-licensed sources, no system tools at runtime".
- [ ] =schemas/manifest.schema.json= : JSON Schema for the manifest
  (every field from the spec).  Validated by both server and client
  test suites.
- [ ] =schemas/api.openapi.yaml= : OpenAPI 3.1 description of the
  v1 endpoints.
- [ ] =server/ota-server.asd=, =admin/ota-admin.asd=, empty packages.
- [ ] =client/go.mod=, three top-level packages (=libota=, =agent=,
  =internal/...=).
- [ ] CI config that builds Lisp and Go, runs an empty test suite,
  cross-compiles the client to Win/macOS/Linux.
- [ ] Reproducible local dev setup : a =Makefile= with =setup=,
  =build=, =test=, =run-server= targets.

** Vendoring

- [ ] =tools/vendor-import.sh= : helper that, given an upstream URL
  and a commit hash, fetches a tarball, extracts under
  =vendor/<name>/= or =client/internal/vendor/<name>/=, copies the
  upstream =LICENSE=, and writes =VENDORED.md= with URL, commit,
  date, and SHA-256 of the imported tree.  Idempotent.  Uses only
  =curl=, =tar=, =sha256sum= — runnable on any developer
  workstation without admin rights.
- [ ] =tools/vendor-verify.sh= : recomputes the SHA-256 of every
  =vendor/*= and =client/internal/vendor/*= subtree and diffs it
  against the recorded value in =docs/dependencies.org=.  Run by
  CI; fails the build on any mismatch.
- [X] Import the three vendored components:
  - [X] =vendor/mendsley-bsdiff/= — BSD-2-Clause, C
  - [X] =vendor/sharplispers-archive/= — BSD-3-Clause, CL
  - [X] =client/internal/vendor/gabstv-go-bsdiff/= — MIT, pure Go
  - Dropped: =jmacd/xdelta= (turned out GPLv2-or-later, see [[file:adr/0004-bsdiff-only-no-xdelta3.org][ADR-0004]]).
  - Dropped: =ably/vcdiff-go= (decoder-only; useless without a
    permissive encoder, see [[file:adr/0004-bsdiff-only-no-xdelta3.org][ADR-0004]]).
- [ ] =docs/dependencies.org= : table of the five vendored
  components, each row carrying: upstream URL, pinned commit hash,
  SPDX licence id, import date, SHA-256 of the imported tree.
- [ ] =docs/THIRD_PARTY_LICENSES.org= : aggregated licence text for
  every vendored component, suitable for ship with the client
  binary (consumed at build time to embed into =ota-agent licenses=
  output).
- [ ] =server/Makefile= target =vendor-build= : builds the C
  =bsdiff= helper (from =vendor/mendsley-bsdiff=) and the C
  =xdelta3= helper (from =vendor/jmacd-xdelta3/xdelta3/=) into
  =server/build/bin/=.  Server boots with the path to those
  binaries baked into config; nothing on the host's =$PATH= is
  consulted.
- [ ] =client/Makefile= target =build= : =go build -trimpath
  -tags=netgo,osusergo= producing a single static binary per OS
  with no =cgo= and no dynamic linkage.

Exit criterion : =make build && make test && make vendor-verify=
passes on a fresh machine without =bsdiff=, =xdelta3=, =tar=
installed system-wide; CI green; the manifest schema validates a
hand-written sample; =ldd ota-agent= (Linux) and =otool -L
ota-agent= (macOS) show no third-party dynamic dependencies.

* Phase 1 — End-to-end happy path on Linux  (3 weeks)

Goal :: a single developer can publish a release from the same Linux
box, and a single client on the same box can install it.  No TLS, no
auth, no patches yet.

** Server (week 1)

- [ ] *Catalogue* : SQLite schema for =Software=, =Release=,
  =Channel=, =Classification=, =Client=, =DownloadEvent= (no
  =Patch= yet).  Migrations are SQL files numbered =0001_…=.
- [ ] *Storage* : content-addressed =blobs/= directory, =put-blob=,
  =get-blob=, =has-blob= primitives.
- [ ] *Deterministic tar writer* : built on top of the vendored
  =sharplispers-archive=; produces the canonical =.tar= as
  specified ([[file:ota-specifications.org::*Blob format and determinism][Blob format and determinism]]).  Property test : tar
  twice, bytes are equal; tar with shuffled directory entries,
  bytes are equal.  If a PAX edge case forces a fallback, that
  fallback is recorded in an ADR and a libarchive FFI shim is
  added to =vendor/=; not expected to be needed.
- [ ] *Manifest* : build + Ed25519-sign + verify.  Signing key is a
  test key on disk for now.
- [ ] *HTTP* : Woo + Clack adapter; routes for =GET /v1/health=,
  =POST /v1/admin/software/{software}/releases= (multipart upload),
  =GET /v1/software/{software}/releases=, =GET .../latest=,
  =GET .../{version}/manifest=, =GET /v1/blobs/{sha256}= (sendfile
  path verified by =strace=).
- [ ] *No auth yet* — a single hardcoded admin token in config.

** Client (week 2)

- [ ] *Manifest verify* : parse JSON, verify Ed25519 signature
  against a pinned public key embedded in the binary.
- [ ] *Transport* : HTTP GET with =Range:= resume, sha256 streaming
  verifier, progress callback.
- [ ] *State* : =$OTA_HOME/<software>/state.json= read/write,
  =current= / =previous= symlinks (junction code stubbed for
  Windows, exercised in phase 3).
- [ ] *Tar extract* : safe extractor (rejects path traversal,
  Windows reserved names, absolute paths, =..= entries).
- [ ] *=ota-agent install <name>=* : full flow, no patches, no
  channels.

** Glue (week 3)

- [ ] *=ota-admin publish <dir> --software=X --version=Y=* against
  the local server.
- [ ] *Integration test harness* : =tests/e2e/install.sh= that
  starts the server in a temp =$OTA_ROOT=, publishes a 50 MB
  release, runs =ota-agent install=, asserts =current= points at
  the right tree, asserts every file's bytes match, kills server
  mid-download and asserts resume works.
- [ ] Telemetry stubs (=POST /v1/events/install=) wired but no-op.

Exit criterion : on a fresh Linux VM, =make e2e= boots a server,
publishes a 50 MB sample, installs it, and the installed tree's
sha256 over deterministic-tar matches the published blob's sha256.

* Phase 2 — Patches  (2 weeks)

Goal :: upgrades transfer only the binary delta; full-blob fallback
exists and is exercised.

- [X] =Patch= entity, schema migration, =patches/= CAS directory.
- [ ] *Patch builder worker* : background pool that consumes a job
  queue (in-memory for now, Postgres-backed in phase 5).  Invokes
  the *vendored* =bsdiff= binary built in phase 0
  (=server/build/bin/bsdiff=) — never the host's =$PATH=.
  =bsdiff= is the only patcher in v1 (see [[file:adr/0004-bsdiff-only-no-xdelta3.org][ADR-0004]]); the worker
  rejects =Software.default_patcher = xdelta3= at publish time.
  Patches are content-addressed by =sha256(patch-bytes)=.
- [ ] *Patch path resolver* : graph search over available patches
  to find the cheapest =from → to= path; includes the full-blob
  fallback ratio (=fallback_ratio=, default 0.7).
- [ ] *=GET /v1/software/{software}/upgrade?from=…&to=…=* returns
  302 to the concrete artefact, or the resolved chain in the JSON
  body if =Accept: application/json=.
- [ ] Manifest =patches_in= is rewritten and re-signed each time a
  patch finishes building.
- [ ] *Client* : =install_or_upgrade= takes the upgrade path,
  applies =bspatch= (vendored =gabstv-go-bsdiff=, in-process,
  pure-Go) to =blob-vX= → =blob-vY=, verifies the result hash
  against the manifest before extracting.  No subprocess, no
  shared library, no host =$PATH= dependency.
- [ ] *Local retention* code in the client : after a successful
  upgrade, drop =vX-2= material per [[file:ota-specifications.org::*Local retention][Local retention]].
- [ ] Integration test : publish three releases, install v1, upgrade
  to v2 (single patch), upgrade to v3 (chained patch), simulate
  missing patch and verify full-blob fallback.

Exit criterion : an upgrade transfer is < 5 % of the full blob size
on the synthetic test (recompiled-binaries-style payload), and the
reconstructed blob hash always matches the manifest.

* Phase 3 — Cross-platform client  (2 weeks)

Goal :: =ota-agent= and =libota= work on Windows and macOS; one CI
job emits all three platform binaries.

- [X] *Symlink/junction abstraction* : =client/internal/platform=
  with =SetCurrent= / =ReadCurrent= / =SwapCurrentToPrevious=.  On
  POSIX it uses =os.Symlink= directly; on Windows it tries the
  symlink (works under developer mode or admin) and falls back to
  a =current.path= text-file shim when symlink creation fails.
  The full junction-via-IOCTL path is deferred — empirically the
  shim is sufficient for non-admin Windows installs, and adding
  =golang.org/x/sys/windows= as a dep can wait until measured need.
- [X] *Path handling* : long-path =\\?\= prefix helper
  (=platform.LongPath=), case-preserving =platform.PathEqual=
  with byte-exact comparison on Linux and case-insensitive on
  macOS / Windows.
- [X] *=libota= as =c-shared=* : =client/cmd/libota-shared= is the
  =package main= that exports =ota_install=, =ota_upgrade=,
  =ota_revert=, =ota_version=, =ota_last_error= via cgo.  Builds
  cleanly with =-buildmode=c-shared= on linux/macOS/windows; the
  generated =libota.h= is the canonical C contract.  =ota-agent=
  remains cgo-free and shares the in-process =package libota=.
- [X] *Cross-compilation matrix* in CI :
  =GOOS=linux,darwin,windows= × =GOARCH=amd64,arm64= already wired
  in phase 0; phase 3 adds the =libota-cshared= cross-OS build job
  and native =go test= matrix on the GitHub mirror.
- [X] Integration tests run on Windows and macOS runners — via the
  GitHub Actions mirror's =go-test= matrix (=ubuntu/macos/windows-
  latest=).  GitLab.com does not provide free Windows/macOS shared
  runners, so the full e2e (which boots the SBCL server) stays
  Linux-only on GitLab; Windows/macOS validation is the unit-test
  layer plus the c-shared build.

Exit criterion : a cold install on Windows 11 from the install page
URL completes without admin rights and without a pre-installed
runtime.

* Phase 4 — Auth, classifications, channels  (2 weeks)

Goal :: the server can decide /who/ sees /what/.

- [X] *TLS* : opt-in via =OTA_TLS_CERT= / =OTA_TLS_KEY= env vars;
  Woo terminates TLS when both are set.  In most deployments TLS
  is terminated by a reverse proxy (kept on the loopback so the
  =sendfile= path stays active).  Hard-required mTLS is deferred —
  a future sub-phase plugs cert-subject identity into
  =resolve-identity=.
- [X] *Bearer-token identity* : a single hardcoded admin token
  authorises admin endpoints; per-client bearers are minted via
  =/v1/exchange-token= against a one-shot install token.
  Cert-subject → client_id resolution is wired in =resolve-identity=
  (=cert-subject= column on =clients=) but not yet activated by an
  mTLS layer.
- [X] *Classifications* : tagged on releases via
  =X-Ota-Classifications= on publish, stored as a JSON column; every
  catalogue read filters by =classification-match-p= over the
  caller's classifications.  E2E proves it: anonymous → 1.0.0;
  =[public,beta]= → 2.0.0-beta.  Channel filtering reuses the same
  mechanism — a separate channel CRUD is deferred (the columns
  already exist; full UI sub-phase later).
- [X] *Install token* : one-shot, short-TTL, =POST
  /v1/admin/install-tokens= (admin), =POST /v1/exchange-token=
  (client).  Agent CLI gains =--install-token= / =--hwinfo=.
  Bearer is persisted in =state.json=.
- [ ] *Rate limits* : deferred to phase 5 (operations) where the
  per-identity token bucket fits with the rest of the operational
  surface.  Note ADR if/when it lands.
- [X] *Audit log* : =append-audit= on every admin write; queryable
  via =GET /v1/admin/audit=.  E2E asserts ≥3 rows after the
  publish + mint flow.

Exit criterion : a release published to channel =canary= is
invisible to a client whose classifications don't include
=canary=, regardless of how the client phrases the query.

* Phase 5 — Operations  (2 weeks)

Goal :: the server is production-shaped.

- [ ] *PostgreSQL backend* — *deferred*.  The catalogue surface
  (=ota-server.catalogue=) is the abstraction line; the v1
  implementation calls =cl-sqlite= directly.  Switching to
  =cl-dbi= behind the same surface is a sub-phase between v1 and
  v1.1; SQLite stays the default until then.  See
  [[file:operations.org::*PostgreSQL catalogue][operations.org]] for the migration plan.
- [ ] *S3 backend* — *deferred*.  The CAS interface
  (=make-cas=, =put-blob-from-file=, =cas-blob-path=,
  =put-patch-from-file=) is structured to admit a second backend;
  only the filesystem implementation exists in v1.  Driven by the
  first customer that needs it.
- [X] *Garbage collection* : =gc-software= prunes releases that
  are not =uncollectable=, not the latest, have ≤ =min_user_count=
  active clients in a configurable window, and are older than
  =min_age_days=.  Exposed at =POST
  /v1/admin/software/{software}/gc= with =dry_run=,
  =min_user_count=, =min_age_days= JSON body.  Idempotent.
- [X] *Verification job* : =verify-storage= walks =blobs/= and
  =patches/=, re-hashes each file, reports mismatches.  Exposed at
  =POST /v1/admin/verify= returning =checked=/=ok=/=bad= triples.
- [X] *Backup/restore* : =tools/backup.sh= tarballs the OTA root
  (db, blobs, patches, manifests, etc/keys), with a SHA-256
  sidecar.  =tools/restore.sh= refuses to clobber a non-empty
  destination.  Verified by =tests/e2e/ops.sh=: backup → restore
  preserves the manifest tree byte-for-byte.
- [X] *Manifest re-sign* : =resign-manifests= rewrites every
  =<sw>/<v>.sig= under =manifests/= with a new keypair without
  touching the .json bytes.  Procedure documented in
  [[file:operations.org::*Key rotation][operations.org → Key rotation]].
- [X] *Operational doc* : [[file:operations.org][docs/operations.org]] — deploy, configure,
  publish, mint install tokens, GC, verify, audit, backup/restore,
  key rotation, sizing, deferred backends, troubleshooting.

Exit criterion : a one-page runbook walks an ops engineer through
deploy, cert rotation, and disaster recovery from a backup, and the
runbook's commands are exercised by an integration test.

* Phase 6 — Recovery tool & polish  (1 week)

Goal :: the recovery story works for users, not just engineers.

- [X] =GET /v1/software/{software}/anchors= : every uncollectable
  release plus the latest visible release; filtered by the
  caller's classifications; newest first.
- [X] =ota-agent doctor <name>= : prints local distributions and
  remote anchors.  =--recover=<version>= installs the chosen
  anchor (multi-step rollback).  Phase-1 install path is reused —
  full-blob download for old anchors, patch path when one exists.
- [X] Admin endpoint to flag a release uncollectable: =POST
  /v1/admin/software/{sw}/releases/{v}/uncollectable=.
- [X] *Reverse-patch generation* — admin endpoint
  =POST /v1/admin/software/{sw}/patches/reverse= builds a Y→X
  patch on demand and adds it to the catalogue (audited).  The
  recovery tool can pull these to downgrade with bandwidth
  savings; if not requested, anchor full-blob downloads still
  work as before.
- [X] *=ota-agent watch=* — background polling daemon.  Long-lived
  foreground process suitable for cron / systemd / launchd /
  Windows Task Scheduler.  =--interval=24h= (default), =--once=
  exits after one check.  Calls =libota.Upgrade= on each tick.
- [X] *Install page* — =GET /v1/install/{software}= renders a
  small HTML page with OS-detected agent download links and the
  one-line install command (the actual token must be embedded by
  the URL the admin sends; the page itself does not mint, since
  it is unauthenticated).  Sufficient as a v1 hand-off shell;
  fancier UX (staggering "your install will start in N minutes")
  is a styling sub-phase.
- [X] *Bulk install-token issuance* — both pieces:
  - =POST /v1/admin/install-tokens/batch= accepts
    ={count, classifications, ttl_seconds}= and returns N tokens
    in one round-trip, capped server-side at 10 000 per call.
  - =ota-admin mint-tokens --csv users.csv [--classifications=...]
    [--ttl=7d] [--output=tokens.tsv]= reads a CSV of identifiers
    and emits a TSV ready for mail merge with one
    =(identifier, classifications, install_token, expires_at,
    install_url)= row each.  TTL accepts =Ns/Nm/Nh/Nd= shorthand.
- [X] *Rate limits per identity* — in-memory token-bucket
  (=app-state-rate-buckets=) keyed on =client_id= for
  authenticated callers and =anon-<remote-addr>= otherwise;
  default capacity 600 tokens, refill 10 tokens/sec.  The =health=
  endpoint is exempt.  Returns 429 with =Retry-After= when
  exhausted.
- [ ] *Mandatory mTLS for admin endpoints* — still deferred.  The
  =cert-subject= column on =clients= is in the schema and
  =resolve-identity= is structured to consume it; what is missing
  is the cl+ssl-side wiring to require + verify a client cert
  on admin paths.  Sub-phase between 1.0 and 1.1.
- [ ] *PostgreSQL backend* — still deferred.  See
  [[file:operations.org::*PostgreSQL catalogue][operations.org]] for the migration plan; the catalogue
  surface is the abstraction line, the work is mechanical
  (cl-sqlite -> cl-dbi) but pervasive (every query function).
- [ ] *S3 backend* — still deferred.  See
  [[file:operations.org::*S3 / S3-compatible blob storage][operations.org]] for the design; the CAS interface admits
  a second backend.

Exit criterion : a user with a broken install can run =ota-agent
doctor <name> --recover=, pick an anchor, and end up on a working
older release without engineer help.

* Phase 7 — Hardening, documentation, 1.0  (1 week)

- [X] *Threat-model walk-through* — [[file:adr/0005-threat-model-walkthrough.org][ADR-0005]] records 20 threats
  with disposition (Closed / Mitigated / Accepted) plus the
  defence in place for each.  Open mitigations (rate limits,
  mandatory mTLS, KMS) are recorded as follow-up items in the
  plan.
- [X] *Fuzz tests* — Go's stdlib =testing.F= harnesses for the
  three client-side attack-surface entry points:
  =manifest.Parse=, =manifest.Verify=, =tarx.Extract=,
  =tarx.safeName=, =patch.Apply=.  Wired into CI via =FUZZTIME=5s
  make fuzz= as a separate verify-stage job.  Burst-tested
  locally to > 1 M execs on the parser, no crashes.

* Cross-cutting concerns

** Telemetry & metrics

- Server : Prometheus =/metrics= endpoint (request counts,
  =sendfile= bytes, patch-build queue depth, GC outcomes, signature
  failures).
- Client : opt-in =POST /v1/events/...= reports as already specified.
- No third-party telemetry; everything stays inside the
  organisation.

** Logging

- Server : structured logs (JSON to stdout), rotated by the OS.
- Client : a small ring buffer in =$OTA_HOME/<software>/logs/= so
  =ota-agent doctor= can show "what happened last time".

** Configuration

- Server : single TOML file at =$OTA_ROOT/etc/ota.toml= as in the
  spec.  No environment-variable mode in v1 — config drift is
  worse than restart-on-change.
- Client : per-user =$OTA_HOME/config.toml=, plus a system-wide
  =/etc/ota/config.toml= (or =%ProgramData%\OTA\config.toml=) for
  shared deployments.

** Build & release of the OTA system itself

- Server : Lisp images built with =sb-ext:save-lisp-and-die=,
  packaged as a tarball + systemd unit.
- Client : Go binaries cross-compiled in one CI job, code-signed
  per OS, uploaded to the install page's CDN.  The aggregated
  =docs/THIRD_PARTY_LICENSES.org= is embedded into the binary at
  build time; =ota-agent licenses= prints it.
- Versions of the OTA system itself are tracked under =VERSION.txt=
  at the repo root and bumped manually at each phase boundary; the
  system's own installer can use =ota-server= to distribute itself
  once phase 7 closes.

** Testing pyramid

- *Unit* : pure functions in =catalogue/=, =manifest/=,
  =transport/=, =patch/=, =tarx/=.  Fast, no I/O.
- *Property* : determinism of the tar writer; round-trip of
  manifest sign/verify; "patch then verify" never produces a
  blob whose hash does not match.
- *Integration* : on-disk e2e tests on each of Linux/macOS/Windows
  in CI.  These are the canaries; they are slow and that is fine.
- *Fuzz* : on inputs from the network (manifest, tar, patch).

* Risks and open items

| Risk                                                         | Mitigation                                                                            |
|--------------------------------------------------------------+---------------------------------------------------------------------------------------|
| Woo's sendfile path silently breaks under TLS                | Phase 7 strace soak; if confirmed broken under TLS, terminate TLS at a reverse proxy that does =sendfile + kTLS= |
| SBCL on Windows for an admin who can't install Lisp          | =ota-admin= Go binary in the alternative-stacks list (spec)                          |
| Ed25519 key rotation breaks pinned clients                   | Initial trust = bundled key list; rotation = signed key-list update over the same channel |
| =bsdiff= memory blow-up on 8 GB blobs                        | Cap blob size at 8 GB; document RAM requirement on the patch worker; consider =bsdiff= replacement (=hdiffpatch=) only if it bites |
| Patch chains get long for users who skipped many releases    | Full-blob fallback ratio (default 0.7); supercheckpoint patches from latest =uncollectable= |
| Windows symlink restrictions in unprivileged contexts        | Junction + launcher shim fallback (phase 3)                                          |
| Need for xdelta3-quality compression on a specific software | bsdiff is the only patcher in v1 (ADR-0004); evaluate =sisong/HDiffPatch= (MIT, C++) if measured savings on a real release set justify the cgo cost.  Customers may also build a private fork that vendors GPL xdelta3 under their own licence terms. |
| Vendored upstream gets a security fix we miss                | =docs/dependencies.org= recorded; quarterly review documented in =docs/operations.org=. |

** ota-agent (client) — subcommands stubbed as "not implemented"

The Go binary's =main.go= advertises six subcommands in =Usage=
that are not wired up; today they print
=ota-agent: "X" not implemented yet= and exit 1
([[file:../client/agent/main.go::64][client/agent/main.go:64]]).

- [ ] *=list [--remote|--local]=* — listing installed software
  (=--local=) and listing what the server has on offer
  (=--remote=).  Local listing is a directory walk under
  =$OTA_HOME/=; remote listing is =GET /v1/software= followed
  by per-software =GET /v1/software/<sw>/manifest=.  Ship target:
  v1.0.x (small).
- [ ] *=show <name>=* — print the active version, server URL,
  classifications, install date, signing key.  Reads
  =$OTA_HOME/<sw>/state.json=; pure-local, no server call.  Ship
  target: v1.0.x.
- [ ] *=verify <name>=* — recompute SHA-256 of every file under
  =$OTA_HOME/<sw>/distribution-<active>/=, compare against the
  manifest, report mismatches.  Operator's first move when an
  install looks corrupted.  Ship target: v1.0.x.
- [ ] *=prune <name> [--archive-depth=N]=* — drop archived
  distribution-* directories beyond the depth, keeping the
  current and (= --archive-depth) most recent for revert.  Frees
  disk on long-lived clients.  Ship target: v1.0.x.

** ota-server — incomplete subcommands and gaps

- [ ] *=ota-server gc= CLI subcommand is a stub*
  ([[file:../server/src/main.lisp::126][main.lisp:126]]).  The actual GC logic is fully
  implemented at the HTTP layer (=POST /v1/admin/software/<sw>/gc=
  with =dry_run / min_user_count / min_age_days= per
  operations.org), but =/opt/ota/ota-server gc= just prints
  "phase-1 stub.".  Wire the subcommand to call into
  =ota-server.workers.operations:run-gc= for an offline /
  cron-friendly invocation that doesn't go through HTTP.  Ship
  target: v1.0.x.
- [ ] *TOML =[server].admin_token= key is not read by the loader.*
  Documented as a gap in operations.org and the CHANGELOG;
  operators must set =OTA_ADMIN_TOKEN= via env (or systemd's
  =EnvironmentFile=).  Wiring it through =load-config-from-file=
  is ~5 lines + a test.  Ship target: v1.0.x.
- [ ] *Real-time publish progress over HTTP (chunked transfer).*
  Today the publish handler's response only arrives once the
  bsdiff fan-in completes; per-patch progress is logged
  server-side but not visible to =ota-admin=.  Switching to a
  chunked response with periodic JSON heartbeats would let the
  client show a progress bar and would also obsolete the
  read-timeout knob added in v1.0.4.  Touches the publish
  handler, the patcher worker (callback hook), and =ota-admin=
  (stream consumer).  Ship target: v1.1.
- [ ] *Async patch-build worker pool (job queue).*  The
  patcher's preamble notes
  ([[file:../server/src/workers/patcher.lisp::7][patcher.lisp:7]]) that the
  proper async worker pool with a Postgres-backed job queue
  lands in phase 5; today the bsdiff fan-in runs synchronously
  in the publish handler thread.  Multi-worker Woo (v1.0.4)
  prevents wedging the server, but a really long publish still
  occupies a worker for the duration.  Ship target: v1.1.

** Backends still on the v1 minimum

Both already documented in operations.org "Backends not yet
implemented":

- [ ] *PostgreSQL catalogue backend.*  The catalogue function
  surface (=ota-server.catalogue=) is stable; the v1
  implementation calls =cl-sqlite= directly.  Adding PG means
  switching to =cl-dbi= behind the same surface.  Migration files
  are already valid SQL on both DBs except for SQLite's
  =strftime= (a per-driver helper hides it).  Ship target:
  sub-phase between v1 and v1.1, driven by first customer
  hitting SQLite's single-writer ceiling.
- [ ] *S3 / S3-compatible blob storage.*  The CAS interface
  (=make-cas=, =put-blob-from-file=, =cas-blob-path=,
  =put-patch-from-file=) admits a second backend; only the
  filesystem implementation exists.  An =s3-cas= variant using
  =zs3= (or shelling out to =aws s3=) is straightforward.  Ship
  target: sub-phase, driven by the first customer hosting on
  cloud object storage.

- [ ] *Mandatory mTLS for admin endpoints

** CI compute optimisation (noticed v1.0.4)

Cutting v1.0.1 → v1.0.4 burned through the Ogamita group's
shared-runner monthly compute allowance fast enough that the
v1.0.4 =release= job sat queued until minutes were topped up.
The two heaviest contributors are the =build-server-image= job
(multi-arch buildx with QEMU-emulated arm64) and the =e2e=
job (sequential SBCL boots per shell test).  Two cheap wins:

- [ ] *Verify the dev-image cache is actually hitting in
  practice.*  =build-dev-image= already declares
  =--cache-from type=registry= and a =rules: changes:= gate on
  =docker/Dockerfile.dev= and =.gitlab-ci.yml=, but the cache hit
  rate isn't measured anywhere.  Run a no-op push, look at the
  buildx output for "CACHED" lines, and confirm we're not
  silently rebuilding layers.  If we are, fix the cache key /
  source ordering.
- [ ] *Gate =build-server-image= on tags only* (drop the
  =$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH= rule).  Today it
  fires on every master push *and* every tag, doubling the
  buildx burn for no benefit -- nothing pulls
  =server:latest= or =server:<short-sha>= operationally; the
  publishable images are the tagged ones.  Cuts a per-pipeline
  ~10 minutes of QEMU time.
