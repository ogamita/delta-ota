#+TITLE: delta-ota — QA plan
#+AUTHOR: Ogamita Ltd.
#+DATE: 2026-05-10
#+OPTIONS: toc:2 num:t

This document defines the QA strategy for Ogamita Delta OTA at
v1.1.0.  It is meant to be read by engineering, by sales (when a
customer asks how the project assures quality), and by reviewers
auditing a release.  Each section lists the *current state*, the
*target*, the *concrete tasks* to close the gap, and the *cadence*
at which the activity runs.

* Aim and scope

The system has three independent shipping artefacts -- =ota-server=
(SBCL on Linux), =ota-admin= (SBCL CLI), and the client (=libota= +
=ota-agent=, Go, multi-OS) -- and a public wire contract
(HTTP/JSON API + manifest schema + libota C ABI + state.json).
The wire contract is frozen for the v1.x line; everything else
evolves.

QA must therefore ensure:

1. *Functional correctness* per release (unit + integration +
   e2e).
2. *Wire-format stability* (cross-version compatibility tests
   between an N-1 client and an N server).
3. *Resilience under stress* (load + soak; the "first-Monday
   morning upgrade storm" sizing in =docs/operations.org= must be
   demonstrable, not asserted).
4. *Defence against malformed / hostile input* (fuzz at every
   parsing boundary).
5. *Auditable change discipline* (code review with an explicit
   checklist, security review for changes that touch the wire,
   the catalogue, or vendored code).

* Test layers — current state

| Layer                         | Tests today | Cadence                |
|-------------------------------+-------------+------------------------|
| Lisp unit (fiveam)            | 82 tests / 141 server checks + 102 admin checks | per-PR via CI =lisp-test= / =lisp-test-admin= |
| Go unit                       | ≈40 test files (libota, listing, verify, manifest, patch, platform, tarx, transport) | per-PR via CI =go-test= |
| Go fuzz                       | 5 targets (manifest Parse / Verify, tarx Extract / SafeName, patch Apply) | per-PR =go-fuzz=, FUZZTIME=5s |
| Shell e2e                     | 5 scripts (run, auth, ops, recovery, parallel) | =make e2e=, run on master + tags + manual |
| CLI smoke                     | =ota-server= + =ota-admin= subprocess tests against the built binary | included in the Lisp test suites |
| Multi-arch build verification | linux / darwin / windows × amd64 / arm64 | per-PR =build-client-*= |
| Self-containment              | =ldd= / =otool -L= ⇒ no third-party dynamic deps | per-PR =self-contained-check= |
| Vendored-source integrity     | SHA-256 re-hash of every vendored subtree | per-PR =vendor-verify= |
| Licence-header sweep          | every source file carries an SPDX header | per-PR =licence-headers= |
| SAST                          | GitLab's Semgrep template | per-PR via =include: SAST= |

What is *not* yet in place:

  - Code coverage measurement (Go and Lisp).
  - Property-based / generative tests beyond the fuzz targets.
  - A documented edge-case catalogue (the cases below exist
    informally as fiveam tests but aren't aggregated).
  - Long-running soak.
  - Cross-version compatibility tests.
  - Performance regression baselines (patch-build time vs blob
    size, install latency vs patch chain length, etc.).
  - The headline load test: 10 000 concurrent clients hitting a
    real Hetzner-hosted server.
  - A formal code-review process with required reviewers,
    security-review triggers, and a PR checklist.

The rest of this document closes those gaps.

* Code review

** Process

- *Two-person rule.* Every change to =server/=, =admin/=,
  =client/libota=, =client/internal/manifest=,
  =client/internal/patch=, =client/internal/transport=, or
  =schemas/= requires a reviewer other than the author before
  merge.  Trivial changes (docs, comments, =.gitignore=, formatting)
  may be self-approved with an explicit "self-merge: docs-only"
  in the merge commit message.
- *Required CI green.* The pipeline (server unit, admin unit,
  client unit + fuzz + cross-compile, e2e on master/tag, vendor
  + licence + SAST) must be green before merge.
- *No force-push to =master=.*  Tags on =master= are immutable;
  releases are cut on =release-X.Y.Z= branches and merged through
  a PR.

** Security-review trigger

A change requires an additional security review (any independent
engineer, recorded as a second sign-off in the PR thread) when
*any* of these are true:

  - touches Ed25519 sign / verify or any crypto path
    (=server/src/manifest/=, =client/internal/manifest=);
  - changes the manifest schema or the HTTP/JSON wire shape
    (=schemas/=);
  - touches authentication / install-token handling
    (=server/src/http/server.lisp= auth helpers,
    =server/src/catalogue/= bearer storage);
  - imports or modifies vendored code (=vendor/=,
    =client/internal/vendor/=);
  - touches the safe-extractor (=client/internal/tarx/=);
  - changes the tarball-determinism path (=server/src/storage/tar.lisp=).

* Edge-case catalogue

A single catalogue lives at =docs/edge-cases.org= (to be created
as part of this plan), one row per case, with three columns:
*description*, *expected behaviour*, *test that covers it*.  No
case in the catalogue is allowed to have an empty *test* column
without an explicit "deferred — issue #N" pointer.

Initial seed (53 cases):

** Manifest / wire

  - empty manifest body
  - manifest with =schema_version: 0= (rejected)
  - manifest with =schema_version: 99= (rejected)
  - manifest with =release_id= mismatch from URL
  - manifest with non-hex blob.sha256
  - manifest signature using a different (untrusted) key
  - manifest signed with the right key but tampered body
  - JSON object key ordering changes ⇒ same manifest_sha256
    (deterministic encoder)

** Patch chain

  - patch where =from_release_id= is unknown
  - patch with unknown =patcher= field
  - patch larger than its target blob (fallback to full blob)
  - patch chain longer than the fallback ratio (default 0.7)
  - apply succeeds but resulting blob's SHA-256 ≠ expected
    (refuse, leave previous current alone)

** Tar / extractor

  - entry with =..= in name
  - entry with NUL in name
  - entry with absolute path
  - entry with Windows reserved name (CON, PRN, …)
  - symlink loop (a → b, b → a)
  - duplicate entries (later wins, no truncation/append)
  - entry with mode 0777 ⇒ normalised to 0644 / 0755
  - 0-byte release
  - 8 GiB release (the documented v1 cap)
  - filenames in CJK / Cyrillic / RTL scripts

** Server (HTTP / catalogue)

  - publish with empty Authorization header
  - publish with the wrong bearer
  - publish with =Content-Length: 0=
  - publish with the wrong content-type
  - re-publish same =(software, os, arch, version)=, same blob
    (200 idempotent — covered by v1.0.4 tests)
  - re-publish same tuple, different blob (409 conflict)
  - publish out-of-order versions (older after newer; latest
    must remain semver-highest)
  - publish two simultaneous within the same wall-clock second
    (published_at strictly monotonic — covered by v1.1.0 tests)
  - manifest fetch for a non-existent release (404)
  - blob fetch with the wrong SHA in the URL (404)
  - blob fetch from a sendfile path while the file is being
    overwritten (refuse during write, serve after rename)
  - install-token claimed twice (second attempt rejected)
  - install-token expired
  - GC with =dry_run=true= (no rows deleted)
  - GC with =min_user_count=0= but =uncollectable= release
    (release retained)

** Client

  - =OTA_HOME= on a read-only filesystem
  - =OTA_HOME= = =$HOME/.ota= but =$HOME= unset
  - Windows symlink fallback to =current.path=
  - SIGINT during blob download (partial blob removed)
  - SIGINT during atomic flip (state stays consistent — old
    current still resolves)
  - state.json that's syntactically valid JSON but missing
    =current= (fail-closed message)
  - state.json that's truncated (fail-closed)

** Configuration / boot

  - server boots with empty config (defaults)
  - server boots with a TOML that contradicts an env-var (env
    wins)
  - server boots with =worker_num=0= (treated as no override)
  - server boots with TOML that has an unknown key (logged,
    ignored)
  - clock jumps backwards mid-boot (catalogue published_at still
    monotonic)

** Operational

  - =systemctl restart= during a publish (publish either
    completes before SIGTERM or 503s cleanly)
  - server killed -9 mid-publish (catalogue + manifests dir
    survives; partial blob removed on next start)
  - disk full on the server during a publish (clean 507; no
    partial release recorded)
  - 10 000 concurrent agent install_event POSTs (record-best-effort,
    no 5xx storm)

** Cadence

- The catalogue is a *checklist*: every PR that adds a behavioural
  branch updates it.  Reviewer enforces.
- A quarterly sweep verifies that every entry's *test* column
  still passes (i.e. the test referenced hasn't been disabled or
  renamed).

* Load testing

** The headline scenario: "first Monday morning"

10 000 unique end-user workstations all wake up at 09:00 local
time on the first Monday of the month.  Each one:

  1. Boots its OS, runs the autostart =ota-agent watch <name>
     --once= command.
  2. Fetches =GET /v1/software/<sw>/releases/latest= (small JSON).
  3. Compares with the locally-installed version; if different,
     proceeds.
  4. Fetches =GET /v1/software/<sw>/releases/<latest>/manifest=
     (small JSON + signature).
  5. Picks a patch from =patches_in= if one exists for its
     current version, else falls back to the full blob.
  6. Downloads the chosen artefact (=GET /v1/blobs/<sha>= or
     =GET /v1/patches/<sha>= — both =sendfile(2)= on the server).
  7. Applies the patch / extracts the blob, atomic flip.

The server's job: serve all that without any client timing out
beyond a generous 10 s p99 wait, and without OOM / file-handle
exhaustion / catalogue lock contention.

** Cadence and cost

- *Per release* (every minor; v1.1, v1.2, …): full L1–L5 run.
  Estimated cost on Hetzner: 1 × AX102 (€105/month, billed
  hourly), 4 × CPX31 (€16/month each, billed hourly), 1 ×
  CPX21 for Grafana (€8/month).  A two-day full sweep is
  roughly €15 in compute.
- *Quarterly*: L6 (24 h soak).  Same infra; ≈ €30 per run.
- *Per patch release* (v1.x.y): a smoke variant (L1 only) on
  Docker locally — no cloud needed.

** Acceptance criteria for shipping a release

A release is *not* tagged until:

  - L1 passes on the same hardware where the previous release
    passed (no regression at the small-fleet end);
  - L4 passes within 10 % of the previous release's metrics
    (no regression at the headline-fleet end);
  - L6 from the previous release is still passing on the
    quarterly schedule (the soak we ship today still works).
