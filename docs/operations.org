#+AUTHOR: Ogamita Ltd.

This is the operator's reference. Day-to-day actions an SRE needs:
deploy, configure, garbage-collect, verify integrity, rotate keys,
back up, restore.

* Deployment topologies

Two supported topologies (see [[file:ota-specifications.org][ota-specifications.org]] for the
architectural detail):

1. *Single dedicated host* — typical for customer-hosted
   deployments.  SQLite catalogue, filesystem blob storage, systemd
   unit, OS-managed TLS certificates.  This is the default.

2. *Kubernetes* — for cloud deployments.  PostgreSQL for the
   catalogue, S3 / S3-compatible (MinIO, AWS S3, …) for blobs and
   patches, the server image scales horizontally behind a load
   balancer.

The same container image (=registry.gitlab.com/ogamita/delta-ota/server=)
serves both topologies; the storage and database backends are
selected in =ota.toml= via the =[storage]= and =[catalogue]= sections.

** Prerequisites

- Linux host (server is Linux-only — Woo + libev).
- For SQLite topology: a writable data directory, the vendored
  =bsdiff= helper at =/opt/ota/bin/bsdiff=.
- For PostgreSQL topology: PG ≥ 14; the catalogue migrations are
  applied automatically on first boot.

** Quick start — container image (canonical)

The server is shipped as a multi-arch container image
(=linux/amd64 + linux/arm64=).  This is the supported production
deployment for both the single-host and Kubernetes topologies.

#+begin_src sh
# 1. pull the tagged image (replace X.Y.Z with the released version)
docker pull registry.gitlab.com/ogamita/delta-ota/server:vX.Y.Z

# 2. prepare the host
sudo mkdir -p /var/lib/ota /etc/ota
sudo install -m 0640 /dev/null /etc/ota/ota.toml
sudoedit /etc/ota/ota.toml             # set hostname, [tls], etc.

# 3. run.  Pass the admin token via -e (or via --env-file).
#    See "Authentication" below -- the same value is what
#    ota-admin must use.
docker run -d --name ota-server                            \
    -p 8443:8443                                           \
    -v /var/lib/ota:/var/lib/ota                           \
    -v /etc/ota:/etc/ota                                   \
    -e OTA_ADMIN_TOKEN="$(openssl rand -hex 32)"           \
    --restart unless-stopped                               \
    registry.gitlab.com/ogamita/delta-ota/server:vX.Y.Z
#+end_src

For Kubernetes, point the container at PostgreSQL and S3 in
=ota.toml= and let the orchestrator manage scheduling — the same
image serves both topologies.

** Quick start — single-host tarball (debug / test / evaluation)

A self-contained tarball, =delta-ota-server-X.Y.Z.tar.gz=, installs
to =/opt/ota= and runs under systemd.  This path exists for
evaluation, debugging, and air-gapped hosts where running a
container is inconvenient.  It is *not* attached to the
GitLab/GitHub Release.

There are two ways to obtain it:

1. *From CI* — the =dist-server= job in =.gitlab-ci.yml= and the
   matching workflow in =.github/workflows/ci.yml= build it on
   =master= and on tags.  Pull it from the pipeline artefacts of
   the tag (or commit) you want.

2. *Locally*, from a checkout of this repository — useful for
   reproducing the artefact a customer is running, or for cutting
   a debug bundle off a feature branch:

   #+begin_src sh
   # Build the SBCL core, vendored helpers, and the tarball.
   # VERSION defaults to `git describe --tags --always --dirty`.
   make dist-server VERSION=1.0.1
   # -> build/dist/delta-ota-server-1.0.1.tar.gz
   # -> build/dist/delta-ota-server-1.0.1/   (unpacked staging dir)
   #+end_src

   This requires the same toolchain as =make build-server=: SBCL +
   Quicklisp at =~/quicklisp= or =/opt/quicklisp=, Go ≥ 1.22, and
   a C compiler for the vendored mendsley bsdiff/bspatch helpers.
   Cross-compiling the tarball for a different host architecture
   is not supported — build on a host of the target arch, or use
   the published container image instead.

The tarball ships a *standalone* =ota-server= executable that bundles
the SBCL runtime; the install host only needs the dynamic-library
runtime deps =libev4=, =libssl3=, and =libsqlite3-0= (Debian package
names).  No system =sbcl= installation is required.

#+begin_src sh
# 1. unpack
sudo mkdir -p /opt/ota /var/lib/ota /etc/ota
sudo tar -C /opt/ota --strip-components=1 -xzf delta-ota-server-X.Y.Z.tar.gz

# 2. user + permissions
sudo groupadd --system ota || true
sudo useradd  --system --gid ota --home-dir /var/lib/ota \
              --shell /usr/sbin/nologin ota || true
sudo chown -R ota:ota /var/lib/ota /etc/ota

# 3. drop in a config
sudo install -m 0640 -o ota -g ota \
     /opt/ota/etc/ota.toml.sample /etc/ota/ota.toml
sudoedit /etc/ota/ota.toml             # set hostname, [tls], etc.

# 4. set the admin token (read by the systemd unit's
#    EnvironmentFile=).  See "Authentication" below -- the same
#    value is what ota-admin must use.
sudo install -m 0640 -o root -g ota /dev/null /etc/ota/environment
echo "OTA_ADMIN_TOKEN=$(openssl rand -hex 32)" \
  | sudo tee -a /etc/ota/environment > /dev/null

# 5. systemd
sudo install -m 0644 /opt/ota/etc/ota-server.service \
     /etc/systemd/system/ota-server.service
sudo systemctl daemon-reload
sudo systemctl enable --now ota-server
#+end_src

* Configuration

Single TOML file (=ota.toml=).  Reload requires a restart for v1.

| Section            | Key             | Default                | Notes |
|--------------------+-----------------+------------------------+-------|
| =[server]=         | =listen=        | =0.0.0.0:8443=         | host:port |
| =[server]=         | =hostname=      | =localhost=            | used in URLs/manifests |
| =[server]=         | =data_dir=      | =/var/lib/ota=         | OTA root |
| =[server]=         | =log_level=     | =info=                 | trace/debug/info/warn |
| =[server]=         | =worker_num=    | =4=                    | Woo HTTP worker threads (each runs its own libev loop) |
| =[tls]=            | =cert=, =key=   | unset                  | optional; otherwise plain HTTP |
| =[catalogue]=      | =db=            | =sqlite:db/ota.db=     | =postgres://...= for PG |
| =[storage]=        | =backend=       | =fs=                   | =fs= (only one in v1) |
| =[patcher]=        | =default=       | =bsdiff=               | bsdiff is the only patcher in v1 (ADR-0004) |
| =[patcher]=        | =bsdiff_path=   | =/opt/ota/bin/bsdiff=  | absolute path; never =$PATH= |
| =[patcher]=        | =worker_count=  | =2=                    | size of the async patch-build worker pool (ADR-0007) |

Environment-variable overrides: =OTA_HOST=, =OTA_PORT=, =OTA_ROOT=,
=OTA_ADMIN_TOKEN=, =OTA_TLS_CERT=, =OTA_TLS_KEY=, =OTA_WORKER_NUM=,
=OTA_PATCH_WORKERS=.
Env-vars take precedence over the TOML file (see
[[file:../server/src/config/loader.lisp][loader.lisp]]).

On startup the server prints its version, the bound address, the
number of HTTP worker threads, and the number of patch-build worker
threads it engaged.  Sample (since v1.2):

#+begin_example
patch-pool: 2 worker threads started
ota-server 1.2.0
listening on 127.0.0.1:8443 (4 http worker threads, 2 patch workers)
data_dir=/var/lib/ota/
manifest pubkey=…
ota-server: ready.
#+end_example

If "(N http worker threads)" reads "(1 http worker thread)" you are
on a single-threaded loop and a long publish will block every other
request -- bump =[server].worker_num= or set =OTA_WORKER_NUM=.

If the boot also prints =patch-pool: reset N stale running jobs
from a previous run= it means the previous process died mid-bsdiff;
those jobs were moved back to =pending= and the new pool will
re-run them.  =bsdiff= is deterministic and the patches table dedups
on (from, to, patcher), so the recovery is idempotent.

* Concurrency model

The HTTP front-end is *multi-threaded* (since v1.0.4): Woo spawns
=[server].worker_num= worker threads, each running its own libev
event loop and sharing the listening socket via accept().  A slow
handler -- typically the synchronous bsdiff patch build during
=publish=, which can take minutes for a 2 GB blob -- occupies one
worker thread for the duration but does *not* block the others.
This is the property =ota-admin= relies on for a sequence of
publishes (and =ota-agent= relies on for first-Monday-morning
upgrade storms): one in-flight publish does not stall concurrent
reads or admin requests.

The catalogue (SQLite) is opened in *WAL mode* with a 10-second
=busy_timeout=, so concurrent readers + one writer never block
each other through the journal.  All catalogue access from the
HTTP layer goes through a recursive lock so the C library is
never re-entered concurrently from the Lisp side either.  The
combination keeps every public endpoint responsive while a long
publish is in flight.

Sizing guidance:

| Workload                            | =worker_num=                                     |
|-------------------------------------+--------------------------------------------------|
| Local development                   | 4 (the dev sample)                               |
| Single-host production, mixed       | ≈ CPU cores; cap by =[patcher].worker_count= + 4 |
| Read-heavy (10 000 clients on first Monday) | 8–16; the =sendfile(2)= fast path scales sub-linearly with worker count |
| Behind a reverse proxy doing TLS    | ≈ proxy concurrency limit                        |

The patcher worker count is *separate*: =[patcher].worker_count=
governs how many concurrent =bsdiff(1)= subprocesses the *async
worker pool* will run (default 2; ADR-0007).  Since v1.2 the
publish handler enqueues one job per prior release into a
persistent queue (=patch_jobs= table) and the pool drains it; the
publish handler itself only tails the queue to emit progress
events on the NDJSON response.  This means a long publish does
*not* pin a Woo HTTP worker for the duration of bsdiff; HTTP and
patch-CPU scale independently.

Tune =worker_count= to roughly =CPU/2= on a dedicated host so the
pool can saturate cores during a publish-storm catch-up without
starving HTTP.  Override per process with =OTA_PATCH_WORKERS=.

* Authentication — the admin token

The =OTA_ADMIN_TOKEN= is a *shared secret*.  The server compares
incoming =Authorization: Bearer <token>= headers against it on every
admin endpoint; =ota-admin= sends exactly the same value to
authenticate its requests.  *The two must match* — the token is
not derived, exchanged, or discoverable; it is whatever the
operator wrote into the server's environment at boot.

** Where the server reads it from

Resolution order, lowest precedence first
([[file:../server/src/config/loader.lisp][server/src/config/loader.lisp]]):

1. *Built-in default* =dev-token= (=*defaults*=).  Convenient for
   local development; *never* acceptable in production.
2. *TOML configuration file*: =[server].admin_token= (since v1.1.1).
   The sample TOMLs ship with the line commented out; uncomment
   and set when env-vars aren't a good fit.  Avoid committing the
   resulting file -- the secret is in plain text.
3. *Environment variable* =OTA_ADMIN_TOKEN= — always wins when set.
   Recommended for production (keeps the secret out of files
   under git or backup).

In practice: env-var first when you can, TOML file when you
must, default only for local dev.

| Deployment       | Where to set =OTA_ADMIN_TOKEN=                                                |
|------------------+-------------------------------------------------------------------------------|
| =make run-server=| Inline on the command line: =OTA_ADMIN_TOKEN=xxxxx make run-server=           |
| Docker / compose | =-e OTA_ADMIN_TOKEN=xxxxx= or an =--env-file=                                 |
| Tarball + systemd| =/etc/ota/environment= (read by the unit's =EnvironmentFile=-/etc/ota/environment=) |
| Kubernetes       | A =Secret= mounted as =OTA_ADMIN_TOKEN= via =envFrom=/=env=                   |

To verify what the running process actually has on a Linux host you
can shell into:

#+begin_src sh
sudo tr '\0' '\n' < /proc/$(pgrep -f ota-server)/environ \
  | grep ^OTA_ADMIN_TOKEN=
#+end_src

** Where =ota-admin= reads it from

=ota-admin= takes the token from the =OTA_ADMIN_TOKEN= environment
variable ([[file:../admin/src/main.lisp][admin/src/main.lisp]]); there is no flag for it (the token
is a credential and command-line history is not the right place
for it).  You typically pull it out of a password store and export
it for the duration of a publish session:

#+begin_src sh
export OTA_SERVER=https://ota.example.com
export OTA_ADMIN_TOKEN=$(pass show ota/admin)
admin/build/ota-admin publish ./build/release/ \
    --software=schms --version=3.4.1          \
    --os=windows    --arch=x86_64
#+end_src

If unset, =ota-admin publish= and =ota-admin mint-tokens= refuse to
proceed with =ota-admin: ota-admin publish: OTA_ADMIN_TOKEN env var
or :token argument required=.

** Timeouts (=OTA_ADMIN_READ_TIMEOUT=, =OTA_ADMIN_CONNECT_TIMEOUT=)

A =publish= triggers a synchronous bsdiff pass on the server: one
patch from every prior release of the same =(software, os, arch)=
to the new one.  For a multi-megabyte blob each patch takes a few
seconds; for a multi-gigabyte blob each can take minutes.  The
=ota-admin= HTTP client must wait for the server to finish before
the 201 response arrives.

Default deadlines (seconds):

| Variable                     | Default | =0= means       |
|------------------------------+---------+-----------------|
| =OTA_ADMIN_READ_TIMEOUT=     | 600     | no read deadline |
| =OTA_ADMIN_CONNECT_TIMEOUT=  | 30      | no connect deadline |

If a =publish= fails with =I/O timeout while reading the response
from URL=, bump =OTA_ADMIN_READ_TIMEOUT= and re-run.  The server's
publish handler is idempotent on the =(software, os, arch,
version)= tuple — a duplicate publish whose first attempt actually
landed on the server (which is what client-side timeouts mean) is
dropped on retry.

#+begin_src sh
# 30-minute deadline for a 4 GiB blob with many priors.
OTA_ADMIN_READ_TIMEOUT=1800 \
  make publish DIR=./build/release SOFTWARE=schms VERSION=3.4.1 \
               OS=windows ARCH=x86_64

# No deadline at all (useful when piping through a slow VPN).
OTA_ADMIN_READ_TIMEOUT=0 \
  admin/build/ota-admin publish ...
#+end_src

** Generating and rotating the token

Generate a fresh token (32 bytes, hex) on the server host:

#+begin_src sh
openssl rand -hex 32
#+end_src

Rotation is a *coordinated edit*: change the value in the server's
environment AND in your password store at the same time, then
restart the server.  Existing per-client install bearers
(=state.json=) are *unaffected* — those are issued by
=POST /v1/exchange-token= and are not derived from the admin
token.  Only operator workflows (=publish=, =mint-tokens=,
=verify=, =gc=, =audit=, =install-tokens/batch=) need the new
admin token.

* Admin cert-subject identity (since v1.4)

The bearer token gives one shared secret all-or-nothing access.
v1.4 adds a second factor for admin endpoints — the verified
client certificate from a reverse proxy — and a strict allowlist
of Subject DNs allowed to use them.  See
[[file:adr/0009-mtls-admin-identity.org][ADR-0009]] for the design.

** What changes for the operator

- Every admin action row in =GET /v1/admin/audit= now carries a
  per-operator =identity= (the cert subject) instead of the
  generic ="admin"=.  Two operators sharing a deployment leave
  distinct traces.
- A leaked =OTA_ADMIN_TOKEN= is no longer sufficient to use
  admin endpoints — the attacker also needs a client
  certificate signed by the proxy's CA and a Subject DN on the
  allowlist.
- With =require_mtls = true=, bearer-only admin requests are
  rejected outright.  This is the strictest deployment, intended
  for "regulated customer" tenancies.

** Configuration

#+begin_src toml
[tls]
# Master switch: ota-server reads the proxy header only when
# this is on.  Off by default to refuse forged subjects on a
# misconfigured deployment.
trust_proxy_subject_header = true

# Which header carries the verified Subject DN.  Defaults to
# x-ota-client-cert-subject; nginx commonly emits x-client-dn
# or ssl-client-s-dn (configure the proxy to set the canonical
# name, or point this knob at whatever the proxy emits).
proxy_subject_header_name  = "x-ota-client-cert-subject"

# Allowlist.  Empty = no enforcement (subject is still recorded
# in the audit log).  Non-empty = strict — only these DNs may
# use admin endpoints.
admin_subjects = [
  "CN=alice@example.com,O=Example",
  "CN=bob@example.com,O=Example",
]

# Strictest mode: bearer alone is not enough.  An admin request
# without a recognised subject is 403.
# require_mtls = true
#+end_src

** Reverse-proxy snippet (nginx)

#+begin_src nginx
location /v1/admin/ {
    ssl_verify_client      on;
    ssl_client_certificate /etc/nginx/clients-ca.crt;
    ssl_verify_depth       3;
    proxy_set_header X-Ota-Client-Cert-Subject $ssl_client_s_dn;
    proxy_pass http://127.0.0.1:8443;
}

location /v1/ {
    # Public read paths -- no client cert needed.
    proxy_pass http://127.0.0.1:8443;
}
#+end_src

* Per-endpoint rate limits (since v1.4)

The v1.0.4 token-bucket limiter (capacity 600, refill 10/sec)
covered every endpoint with one bucket per identity.  v1.4 keys
the bucket on =(identity, endpoint)= and applies stricter
defaults to admin routes.  Non-admin routes keep the v1.3
behaviour byte-for-byte.

Built-in admin caps (in tokens; refill in tokens/sec):

| Route                              | Cap | Refill    | Steady rate |
|------------------------------------+-----+-----------+-------------|
| =admin-publish-release=            | 10  | 0.166/sec | ~10/min     |
| =admin-mint-install-token=         | 60  | 1   /sec  | 1/sec       |
| =admin-mint-install-tokens-batch=  | 5   | 0.0166/sec | 5/hour    |
| =admin-build-reverse-patch=        | 10  | 0.166/sec | ~10/min     |
| =admin-mark-uncollectable=         | 30  | 0.5 /sec  | ~30/min     |
| =admin-gc=                         | 6   | 0.0166/sec | 6/hour    |
| =admin-verify=                     | 6   | 0.0166/sec | 6/hour    |

Honest CI publishing 3-4 releases per day per software line is
two orders of magnitude under the publish cap.  A hostile admin
token hits 429 in seconds.

** Operator override

#+begin_src toml
[rate_limits]
# Format: "CAPACITY/REFILL-PER-SEC".  Fractional refill OK.
admin-publish-release          = "30/0.5"     # 30/min for a launch day
admin-gc                       = "2/0.0000231"  # 2/day
#+end_src

A rate-limited admin request gets =429 Too Many Requests=
with =Retry-After: 1=.  =ota-admin= will surface that as
a friendly retry message.

* Day-to-day operations

** Publish a release

=ota-admin= is built from this repository as a standalone executable
at =admin/build/ota-admin=:

#+begin_src sh
make build-admin   # produces admin/build/ota-admin
#+end_src

From a developer workstation.  =OTA_ADMIN_TOKEN= must match the value
the server was started with — see [[*Authentication — the admin token][Authentication — the admin token]].

#+begin_src sh
export OTA_SERVER=https://ota.example.com
export OTA_ADMIN_TOKEN=$(pass show ota/admin)

# Either invoke the binary directly...
admin/build/ota-admin publish ./build/release/  \
    --software=schms                            \
    --version=3.4.1                             \
    --os=windows --arch=x86_64                  \
    --os-versions=10,11

# ...or use the Makefile convenience wrapper:
make publish DIR=./build/release SOFTWARE=schms VERSION=3.4.1 \
             OS=windows ARCH=x86_64 OS_VERSIONS=10,11
#+end_src

The server hashes the payload, stores the blob, builds patches from
every prior release of the same =(software, os, arch)= triple, and
re-signs the manifest.  Per-patch progress is logged to the
server's stdout ("publish: bsdiff N/M from VERSION ..."), so an
operator tailing the log can see how far through the fan-in pass
the publish is.

The publish handler is *idempotent* on the
=(software, os, arch, version)= tuple:

| Repeat call shape                                | Server response |
|-------------------------------------------------+-----------------|
| Same tuple, same blob hash                       | =200= with =idempotent: true= -- the prior 201 is what's on the server, the patches it built are still there |
| Same tuple, *different* blob hash                | =409 Conflict= -- bump the version, or delete the existing release first |
| New tuple (any of software/os/arch/version differs) | =201= as usual  |

This is what makes a client-side I/O timeout (see Authentication
=> Timeouts above) a safe retry: re-running the same publish after
the first attempt landed costs you a re-upload but does not
duplicate or corrupt anything.

** Mint an install token

Required for first-time installs by an unmanaged client.

For a single token, =curl= is enough:

#+begin_src sh
curl -X POST                                        \
     -H "Authorization: Bearer $OTA_ADMIN_TOKEN"    \
     -H "Content-Type: application/json"            \
     -d '{"classifications":["public"],"ttl_seconds":900}'  \
     https://ota.example.com/v1/admin/install-tokens
#+end_src

For bulk minting from a CSV of users (one row per token, name +
email + classification), use the admin CLI -- it emits a TSV ready
for mail-merge:

#+begin_src sh
admin/build/ota-admin mint-tokens             \
    --csv=users.csv                           \
    --classifications=stable                  \
    --ttl=7d                                  \
    --output=tokens.tsv

# Or via the Makefile wrapper:
make mint-tokens CSV=users.csv CLASSIFICATIONS=stable TTL=7d
#+end_src

The token is one-shot and short-TTL.  The user pastes it into
=ota-agent install ... --install-token=<token>=, which exchanges it
for a per-workstation bearer that is persisted in =state.json=.

** End-user inspection commands (=ota-agent= -- v1.1.0+)

These are read-only / disk-management commands the end user can
run against their own =$OTA_HOME= without involving the server
(or, with =--remote=, against the server in read-only mode).

| Command                              | What it does                                                                               |
|-------------------------------------+--------------------------------------------------------------------------------------------|
| =ota-agent list=                     | Table of every software installed under =$OTA_HOME=: current / previous / on-disk size / server URL |
| =ota-agent list --remote --server URL= | Every release (one row per (software, version) pair) on the server.  Add =--latest= for one row per software with its latest version. |
| =ota-agent show <name>=              | Per-software details: versions, server URL, last-updated, every =distribution-*= directory (=*= active, =p= previous) |
| =ota-agent verify <name>=            | Two-phase integrity check: state file + current symlink + distribution dir, then signed manifest fetch + on-disk blob SHA-256 vs manifest |
| =ota-agent verify <name> --offline=  | Skip the network phase (no manifest fetch)                                                 |
| =ota-agent prune <name>=             | Delete old =distribution-*= directories.  Always keeps =current= and =previous=; =--archive-depth=N= keeps =N= most-recent extras (default 2); =--dry-run= reports without deleting |

These commands are pure read-only on the server side (=list
--remote= calls =GET /v1/software= and the latest-release
endpoint); =verify= additionally fetches a signed manifest.  They
do not require =OTA_ADMIN_TOKEN=.

For unattended dashboards / orchestration, every command exits
non-zero on failure and writes its data to stdout.

The "latest release" semantics changed in v1.1.0 — see
[[*Latest-release semantics][Latest-release semantics]] below.

** Latest-release semantics

The server endpoint =GET /v1/software/<sw>/releases/latest= drives
=ota-agent install/upgrade/watch= when no explicit =--version= is
given, and =ota-agent list --remote --latest=.  Its definition of
"latest" changed between v1.0.x and v1.1.0:

| Server | "latest" =                                                                                |
|--------+--------------------------------------------------------------------------------------------|
| v1.0.x | most recently *published* release (=ORDER BY published_at DESC=)                          |
| v1.1.0 | highest *semver* version, with most-recently-published as fallback when no version parses |

Why the change: under v1.0.x, an operator who re-published a
hotfix to an older version after a newer one was already out
would silently see =watch= *downgrade* every installed client to
the hotfix.  v1.1.0 makes "latest" mean what an operator expects:
the highest version number.

Semver rules (subset, see
[[file:../server/src/catalogue/db.lisp][catalogue/db.lisp]]):

  - =MAJOR.MINOR.PATCH= comparison is per-component numeric
    (so =1.0.10= > =1.0.9=, not the other way round).
  - =MAJOR.MINOR.PATCH-PRERELEASE= sorts *less than* the same
    triple without a prerelease tag (so =1.0.0-rc1= < =1.0.0=).
  - Versions whose components are not all integers (e.g.
    =alpha=, =1.x.0=) do not parse as semver and are ignored
    when selecting "latest" -- as long as there is at least one
    parseable version.  When *no* version parses, the v1.0.x
    fallback applies.

Date-based version strings like =2026.05.10= parse as semver
(all components are integers) and order correctly.

If you want the v1.0.x semantics back -- e.g. you have a
deliberate hotfix-to-older-version workflow -- run a custom
patch level or sort-by =published_at= using
=GET /v1/software/<sw>/releases= directly (which still returns
all rows in =published_at DESC= order).

** Garbage-collect old releases

Drops releases that are not =uncollectable=, not the latest in their
software, have ≤ =min_user_count= clients in the recency window, and
have been published longer than =min_age_days=.

#+begin_src sh
# Dry-run first.
curl -X POST                                                 \
     -H "Authorization: Bearer $OTA_ADMIN_TOKEN"             \
     -H "Content-Type: application/json"                     \
     -d '{"dry_run": true, "min_user_count": 0, "min_age_days": 30}'  \
     https://ota.example.com/v1/admin/software/schms/gc

# Then for real.
curl -X POST                                                 \
     -H "Authorization: Bearer $OTA_ADMIN_TOKEN"             \
     -H "Content-Type: application/json"                     \
     -d '{"min_user_count": 0, "min_age_days": 30}'          \
     https://ota.example.com/v1/admin/software/schms/gc
#+end_src

GC is idempotent and safe to schedule from cron / systemd timer.
The blob is only deleted when no surviving release references it.

** Verify storage integrity

Walks every blob and patch on disk, recomputes SHA-256, reports
mismatches.  Run from cron weekly.

#+begin_src sh
curl -X POST                                                 \
     -H "Authorization: Bearer $OTA_ADMIN_TOKEN"             \
     https://ota.example.com/v1/admin/verify
#+end_src

The response carries =checked=, =ok=, and =bad= (a list of
=(path, actual, expected)= triples).  Any =bad= entry is a
corrupted artefact and warrants an investigation /immediately/ —
likely disk rot or a botched manual edit.

** Inspect the audit log

#+begin_src sh
curl -H "Authorization: Bearer $OTA_ADMIN_TOKEN" \
     https://ota.example.com/v1/admin/audit
#+end_src

Each row records the identity (admin / client_id / "gc"), the
action, the target, optional detail, and a UTC timestamp.

* Backup and disaster recovery

** Take a backup

The server can keep running.  SQLite's WAL means concurrent reads
are fine; the resulting backup is a consistent snapshot at the
moment =cp= reaches the DB file.

#+begin_src sh
sudo tools/backup.sh /var/lib/ota /var/backups/ota
# -> /var/backups/ota/ota-backup-20260501T184500Z.tar.gz
# -> /var/backups/ota/ota-backup-20260501T184500Z.tar.gz.sha256
#+end_src

For PostgreSQL deployments, also run =pg_dump= and add the dump to
the tarball.  S3-backed storage doesn't need to be in the tarball —
the bucket itself is the backup, replicated by the cloud provider.

** Restore from a backup

Stops the world for the time it takes to untar.  The destination
must be empty:

#+begin_src sh
sudo systemctl stop ota-server
sudo tools/restore.sh ota-backup-20260501T184500Z.tar.gz /var/lib/ota
sudo systemctl start ota-server

# Verify integrity once the server is back up.
curl -X POST -H "Authorization: Bearer $OTA_ADMIN_TOKEN" \
     https://ota.example.com/v1/admin/verify
#+end_src

* Key rotation

The Ed25519 manifest-signing key lives at =$OTA_ROOT/etc/keys/=.
Rotation procedure:

1. Generate the new keypair (offline, on a hardened workstation):
   #+begin_src sh
   sbcl --non-interactive --no-userinit --no-sysinit \
        --load ~/quicklisp/setup.lisp \
        --eval '(asdf:load-asd (truename "server/ota-server.asd"))' \
        --eval '(ql:quickload "ota-server" :silent t)' \
        --eval '(let ((kp (ota-server.manifest:load-or-generate-keypair "/tmp/newkey/")))
                  (format t "PUB=~A~%" (ota-server.manifest:keypair-public-hex kp)))'
   #+end_src

2. Distribute the *new public key* to clients via your existing
   trust-bundle channel /before/ rotating.  Until clients trust the
   new key, they will refuse manifests signed with it.

3. Stop the server, replace =ed25519.priv= and =ed25519.pub= in
   =$OTA_ROOT/etc/keys/=, restart.

4. Re-sign every manifest in place (no manifest bytes change; only
   the .sig files):
   #+begin_src sh
   sbcl --non-interactive --no-userinit --no-sysinit \
        --load ~/quicklisp/setup.lisp \
        --eval '(asdf:load-asd (truename "server/ota-server.asd"))' \
        --eval '(ql:quickload "ota-server" :silent t)' \
        --eval '(let ((kp (ota-server.manifest:load-or-generate-keypair
                            "/var/lib/ota/etc/keys/")))
                  (ota-server.workers:resign-manifests
                   "/var/lib/ota/manifests/" kp))'
   #+end_src

5. Once your monitoring confirms all active clients have accepted
   the new key, retire the old key.

* TLS certificate rotation

The recommended deployment terminates TLS at a reverse proxy
(nginx, HAProxy, or a cloud LB).  When TLS is offloaded the rotation
procedure is the proxy's, not the OTA server's.

When TLS is terminated by Woo directly (=OTA_TLS_CERT= and
=OTA_TLS_KEY= set):

1. Drop the new cert/key into the configured paths.
2. =systemctl restart ota-server=.

* Sizing and tuning

Steady-state numbers from the spec
([[file:ota-specifications.org::*Performance, sizing and operational limits][Performance, sizing and operational limits]]):

- 2 GB blob, monthly cadence, mostly recompiled binaries → median
  patch size ≈ 20 MB (5–50 MB range).
- 10 000 clients all upgrading at 09:00 Monday → 200 GB egress.
  At 2 × 10 Gb/s the server saturates for ~80 s — acceptable
  without explicit staggering.
- Cold install of a 2 GB blob × 10 000 first-time clients → 20 TB
  egress.  *This needs staggering* — install tokens are
  rate-limited at the install page (token-bucket per channel).

* Backends not yet implemented

** PostgreSQL catalogue

Phase-5 abstracts the catalogue behind a clean function surface
(=ota-server.catalogue=) but the v1 implementation calls
=cl-sqlite= directly.  Adding PG means switching to =cl-dbi= behind
the same surface; the migration files stay valid SQL on both DBs
(SQLite's =strftime= is the only divergence — a per-driver helper
will hide it).  Ship target: a sub-phase between v1 and v1.1.

** S3 / S3-compatible blob storage

The CAS interface (=make-cas=, =put-blob-from-file=,
=cas-blob-path=, =put-patch-from-file=) is structured to admit a
second backend, but only the filesystem implementation exists in
v1.  An =s3-cas= variant using =zs3= or shell-out to =aws s3= is
straightforward to add.  Ship target: a sub-phase, driven by the
first customer who needs it.

* Fleet statistics (since v1.5)

The =client_software_state= snapshot table tells the server who
is on what release right now (per (client, software)).  =ota-
agent= reports the snapshot after every successful install /
upgrade / revert / recover / uninstall, so the GC's "is anyone
still using this release?" check is exact instead of approximate.
ADR-0010 has the full design.

Operationally this means:

- =POST /v1/admin/software/<sw>/gc=  with =min_user_count > 0=
  now actually protects in-use releases.  Pre-v1.5 the threshold
  was structurally off because no agent ever reported.
- =GET /v1/admin/stats/...= exposes a curated catalogue of
  queries against the snapshot + the event log.
- =ota-server stats <name> [flags]= is the cron-friendly mirror
  of the HTTP surface.

** The query catalogue

| Name                     | Required params      | Optional        | Returns                          |
|--------------------------+----------------------+-----------------+----------------------------------|
| =population-per-release= | =software=           | -               | (release_id, clients)            |
| =fleet-summary=          | -                    | -               | (software, clients, distinct_releases, most_recent_update) |
| =stale-clients=          | =software=           | =since-days=30= | (client_id, current_release_id, last_updated_at) |
| =gc-impact=              | =software=           | -               | (release_id, version, published_at, uncollectable, clients) |
| =recent-events=          | =software=           | =since-days=30= | (day, kind, status, n)           |
| =upgrade-failure-rate=   | =software=           | =since-days=30= | (from_release_id, to_release_id, failures, attempts, failure_pct) |
| =adoption-curve=         | =software=, =release-id= | -           | (day, adopted_today, total)      |
| =recovery-events=        | =software=           | =since-days=30= | (day, from_release_id, to_release_id, status, n) |

=GET /v1/admin/stats= (no path component) lists every available
query with its parameters and result schema — useful when an
admin tool wants to render a dynamic UI.

Per-row result cap: 1000.  Operators wanting fuller dumps run
=sqlite3 db/ota.db= directly (single-host deployment) or the
PostgreSQL CLI (when that backend ships).

** Audit trail

Every stats query is logged with =action = "stats-run"=,
=target = <query-name>=, =detail = <query-string>= in the v1.4
admin audit log.  Operators can prove who looked at fleet data
and when.

** Agent-side opt-out

A deployment that doesn't want the server to know which release
each install is running can set =OTA_DISABLE_REPORTING=1= in
the agent's environment.  The local install proceeds normally;
only the server-side snapshot row is suppressed.  =count-
users-at-release= then returns 0 for that client's release, so
the GC's =min_user_count= protection is structurally off for
opted-out clients — operators should pair this with
=uncollectable= pins on releases they want to keep.

* Resumable downloads (since v1.3)

The server honours =Range: bytes=N-[M]= on =/v1/blobs/<sha>= and
=/v1/patches/<sha>=, replying =206 Partial Content= with the
requested slice; full responses advertise =Accept-Ranges: bytes=.
=ota-agent= keeps its =<dst>.part= file across attempts: on retry
it scans the trailing-zero tail (a torn write may have grown the
file's apparent size beyond what was committed to disk),
truncates to the last non-zero byte, recomputes the prefix SHA,
and asks the server for =bytes=N-=.  If the server (or an
intermediate proxy) ignores the Range and replies 200, the agent
falls back to a clean restart from byte 0.  See
[[file:adr/0008-resumable-blob-downloads.org][ADR-0008]] for the
full design.

Operational implications:

- =.part= files live next to the final destination
  (=<OTA_HOME>/<software>/blobs/<sha>.part=).  An aborted install
  leaves them on disk; the next install resumes from there, or
  the operator can delete them manually to force a fresh pull.
- No new configuration knobs.  Range support is always on; resume
  is always on.  The trailing-zero scan window (16 MiB) is a
  compile-time constant in =transport.go=.
- A reverse proxy in front of =ota-server= (nginx, haproxy) MUST
  pass =Range:= through and not buffer 206 responses, or it will
  defeat the optimisation.  All commodity proxies do this by
  default; only check if you've explicitly disabled =proxy_buffering=
  or set =proxy_force_ranges off= or equivalent.

* Troubleshooting

| Symptom                                         | Likely cause                                     | First action |
|-------------------------------------------------+--------------------------------------------------+--------------|
| Client gets 404 on a release that was published | Classification mismatch                          | Verify install token's classifications include the release's |
| =manifest fetch: signature does not verify=     | Server's signing key was rotated; client has stale public key | Push new public key to clients |
| =bsdiff= subprocess fails on publish            | Blob exceeds available RAM (= 2 × blob size)     | Stage the patch worker on a host with more RAM, or use =uncollectable= so the build is offline |
| GC pruned a release a customer still uses       | Telemetry incomplete (no install_event)          | Re-publish the release; investigate why telemetry was missing |
| =sendfile= path silently breaks under TLS       | TLS encryption happens in userspace              | Move TLS termination to a reverse proxy with kTLS |
| Big initial install never completes over flaky link | Reverse proxy stripping =Range:= headers      | Test directly against the server: =curl -I -H 'Range: bytes=0-99' .../v1/blobs/<sha>=; expect 206 |
| =.part= file accumulating disk space             | Aborted install never resumed                   | Delete =<OTA_HOME>/<sw>/blobs/<sha>.part= or run =ota-agent install <sw>= to complete it |
| Admin call returns 403 "subject not authorised"  | Operator's cert DN not in =[tls].admin_subjects= | Add the DN to the allowlist (or remove =admin_subjects= entirely to disable enforcement) |
| Admin call returns 403 "client cert required"    | =[tls].require_mtls = true= but no cert reached the server | Verify the reverse proxy is requesting + verifying client certs AND injecting the configured header |
| Admin operations hit 429 unexpectedly             | Per-endpoint cap is tighter than the workload   | Set =[rate_limits].<endpoint> = "C/R"= to raise the cap; see ADR-0009 |
| GC skips releases that have no real users         | Snapshot empty (pre-v1.5 agents still in fleet) | Set =min_user_count=0= until the fleet finishes upgrading to v1.5 agents |
| =count-users-at-release= reports 0 for known-active release | =OTA_DISABLE_REPORTING=1= on that client's agent | Pin the release as =uncollectable= or accept that opted-out clients don't count toward the threshold |
